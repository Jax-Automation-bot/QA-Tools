# LocalFileTool 媒体哈希变更 —— 逻辑文档

> 作用：扫描指定目录下的图片 / 视频文件，计算它们的 MD5、SHA-256，并通过在文件尾部追加随机字节，
> 使每次执行后这两个哈希值都和上一次不同（同时保证文件体积不会无限增长）。

---

## 1. 整体架构（三层 + 一个数据载体）

```
HTTP 请求 (POST /api/media/mutate?path=...)
        │
        ▼
┌─────────────────────────┐
│ ApiController           │  ← 接口层（Web 入口）
│  controller 包          │
└───────────┬─────────────┘
            │ 调用 mutateAll(path)
            ▼
┌─────────────────────────┐
│ FileMd5Hash             │  ← 业务层（核心脚本 / 服务）
│  script 包              │
└───────────┬─────────────┘
            │ 每个文件产出一个
            ▼
┌─────────────────────────┐
│ MediaHashResult         │  ← 数据载体（DTO，记录前后哈希）
│  model 包               │
└─────────────────────────┘

LocalFileToolApplication  ← 启动类（Spring Boot 入口，负责装配上面所有 Bean）
FileMd5HashTest           ← 测试类（手动跑真实目录，验证逻辑）
```

| 角色 | 类 | 包 | 职责一句话 |
|---|---|---|---|
| Web 入口 | `ApiController` | `controller` | 接收 HTTP 请求，把路径转交给业务层 |
| 业务核心 | `FileMd5Hash` | `script` | 扫描目录、算哈希、改文件、回填结果 |
| 数据载体 | `MediaHashResult` | `model` | 装一个文件「变更前 / 变更后」的对比数据 |
| 启动类 | `LocalFileToolApplication` | 根包 | 启动 Spring，自动装配并注入上面的 Bean |
| 测试类 | `FileMd5HashTest` | `script`(test) | 手动对真实目录跑一遍并打印结果 |

## 2. 调用链路总览（谁调用谁）

```
[调用方A：HTTP 客户端 / Postman / 前端]
   └─ POST /api/media/mutate?path=D:/photos
        └─ ApiController.mutate(path)                     [接口层]
             └─ FileMd5Hash.mutateAll(path)              [业务层入口]
                  ├─ classify(fileName)                  [私有：判断是不是媒体文件]
                  └─ mutateOne(file, type)               [私有：处理单个文件]
                       ├─ stripTrailer(before)           [私有：剥掉上次追加的尾块]
                       ├─ appendTrailer(base)            [私有：追加新的随机尾块]
                       └─ MediaHashResult.builder()...   [构造结果对象]

[调用方B：FileMd5HashTest（手动测试，不经过 HTTP）]
   └─ FileMd5HashTest.mutateRealDirectory()
        └─ FileMd5Hash.mutateAll(REAL_PATH)              [直接 new 一个 FileMd5Hash 调用]
             └─（同上）
```

**两个入口**都最终汇聚到 `FileMd5Hash.mutateAll(...)`：
- 入口 A：线上 / 联调用 —— 通过 Controller 走 HTTP。
- 入口 B：本地验证用 —— 测试类直接 `new FileMd5Hash()` 调用，不启动 Web。

## 3. 逐个类 / 方法说明

### 3.1 `ApiController`（接口层，`controller` 包）

| 方法 | 签名 | 做什么 | 谁调用它 | 它调用谁 | 用到的依赖 |
|---|---|---|---|---|---|
| `mutate` | `List<MediaHashResult> mutate(@RequestParam String path)` | 接收 HTTP 请求，记录日志，把 `path` 透传给业务层并返回结果列表（自动序列化成 JSON） | **外部 HTTP 客户端**（Postman / 前端 / curl），由 Spring MVC 路由触发 | `FileMd5Hash.mutateAll(path)` | spring-boot-starter-web（`@RestController`/`@PostMapping`/`@RequestParam`）、Lombok（`@Slf4j` 日志、`@RequiredArgsConstructor` 构造注入） |

- **何时被调用**：应用启动后，每收到一次 `POST /api/media/mutate?path=xxx` 就触发一次。
- **依赖怎么来的**：`private final FileMd5Hash fileMd5Hash;` 由 Lombok 的 `@RequiredArgsConstructor` 生成构造方法，Spring 在启动时自动注入（因为 `FileMd5Hash` 标了 `@Service`，是个 Bean）。
- **返回值**：`List<MediaHashResult>`，Spring 的 Jackson 自动转成 JSON 返回给调用方。

### 3.2 `FileMd5Hash`（业务核心，`script` 包，标注 `@Service`）

这是整个工具的心脏。一个 `public` 入口方法 + 四个 `private` 辅助方法。

#### ① `mutateAll(String dirPath)` —— 唯一公开入口

| 项 | 说明 |
|---|---|
| 做什么 | ①校验路径存在且是目录；②递归列出目录下所有文件；③逐个判断是不是媒体文件；④是则交给 `mutateOne` 处理；⑤汇总并打日志返回 |
| 谁调用它 | `ApiController.mutate()`（线上）、`FileMd5HashTest.mutateRealDirectory()`（测试） |
| 它调用谁 | `classify(...)`、`mutateOne(...)` |
| 用到的依赖 | **commons-io** 的 `FileUtils.listFiles(root, null, true)` 递归列文件；Lombok `@Slf4j` 打日志；JDK `java.io.File` |
| 失败处理 | 路径不存在 / 不是目录 → 抛 `IllegalArgumentException` |

执行内部顺序：
```
new File(dirPath)
  → 不存在 / 不是目录 → 抛 IllegalArgumentException
  → FileUtils.listFiles(..., 递归=true)  得到所有文件
  → for 每个 file:
        type = classify(file名)
        type == null（非媒体）→ 跳过
        否则 results.add( mutateOne(file, type) )
  → log.info 统计「共处理 N 个，成功 M 个」
  → return results
```

#### ② `mutateOne(File file, String type)` —— 处理单个文件（private）

| 项 | 说明 |
|---|---|
| 做什么 | 算旧哈希 → 剥旧尾块 → 追加新随机尾块（循环确保新 MD5 ≠ 旧 MD5）→ 写回文件 → 算新哈希 → 封装成 `MediaHashResult` |
| 谁调用它 | `mutateAll(...)` |
| 它调用谁 | `stripTrailer(...)`、`appendTrailer(...)`、`MediaHashResult.builder()` |
| 用到的依赖 | **commons-codec** 的 `DigestUtils.md5Hex / sha256Hex` 算哈希；**commons-io** 的 `FileUtils.readFileToByteArray / writeByteArrayToFile` 读写文件；Lombok 生成的 `MediaHashResult` builder；Lombok `@Slf4j` 日志 |
| 失败处理 | 读写出 `IOException` → 不抛，转成 `changed=false` + `error=信息` 记录在结果里（单个文件失败不影响其它文件） |

关键循环（保证「每次都不一样」）：
```java
do {
    mutated = appendTrailer(base);      // 追加 16 字节随机数 + 魔数
    newMd5  = DigestUtils.md5Hex(mutated);
} while (newMd5.equals(oldMd5));        // 极小概率撞上旧值就重试
```

#### ③ `stripTrailer(byte[] data)` —— 剥掉上次追加的尾块（private）

| 项 | 说明 |
|---|---|
| 做什么 | 检查文件末尾 8 字节是不是魔数 `JAXMUT01`；是则把最后 `24` 字节（16 随机 + 8 魔数）切掉，返回原始内容；不是则原样返回 |
| 为什么存在 | 防止文件体积随每次执行无限增长 —— 每次都「先减后加」，永远只保留一个尾块 |
| 谁调用它 | `mutateOne(...)` |
| 它调用谁 | 仅 JDK：`Arrays.copyOf(...)` |
| 用到的依赖 | 无第三方，纯 JDK |

#### ④ `appendTrailer(byte[] base)` —— 追加新的随机尾块（private）

| 项 | 说明 |
|---|---|
| 做什么 | 生成 16 字节随机数（`SecureRandom`），在原始内容后拼上 `[随机负载 + 魔数]`，返回新字节数组 |
| 为什么能改变哈希 | 内容变了 → MD5 / SHA-256 必然变；随机数保证和上次不同 |
| 谁调用它 | `mutateOne(...)` |
| 它调用谁 | JDK：`SecureRandom.nextBytes(...)`、`System.arraycopy(...)` |
| 用到的依赖 | 无第三方，纯 JDK（`java.security.SecureRandom`） |

> 兼容性：追加的字节在图片 / 视频的有效数据之后（EOI / IEND / moov 之后），JPEG/PNG/GIF/MP4 等查看器、播放器会忽略，文件仍可正常打开。

#### ⑤ `classify(String fileName)` —— 判断文件类型（private）

| 项 | 说明 |
|---|---|
| 做什么 | 取扩展名转小写，命中图片集合返回 `"IMAGE"`，命中视频集合返回 `"VIDEO"`，都不命中返回 `null`（= 非媒体，跳过） |
| 谁调用它 | `mutateAll(...)` |
| 它调用谁 | **commons-io** 的 `FilenameUtils.getExtension(...)` |
| 用到的依赖 | commons-io（取扩展名） |
| 识别范围 | 图片：jpg/jpeg/png/gif/bmp/webp/tif/tiff/heic/heif；视频：mp4/avi/mov/mkv/flv/wmv/webm/m4v/mpeg/mpg/3gp/ts |

### 3.3 `MediaHashResult`（数据载体，`model` 包）

| 项 | 说明 |
|---|---|
| 是什么 | 一个纯数据对象（DTO），装单个文件「变更前 / 变更后」的对比信息 |
| 字段 | `filePath`、`type`、`oldSize`/`newSize`、`oldMd5`/`newMd5`、`oldSha256`/`newSha256`、`changed`、`error` |
| 谁创建它 | `FileMd5Hash.mutateOne(...)` 用 `MediaHashResult.builder()...build()` 构造 |
| 谁消费它 | `ApiController` 把它（列表）序列化成 JSON 返回；`FileMd5HashTest` 打印它 |
| 用到的依赖 | Lombok：`@Data`（自动生成 getter/setter/toString 等）、`@Builder`（自动生成链式 builder） |

### 3.4 `LocalFileToolApplication`（启动类，根包）

| 项 | 说明 |
|---|---|
| 做什么 | Spring Boot 启动入口（`main` 方法），`@SpringBootApplication` 触发组件扫描，自动发现并装配 `ApiController`、`FileMd5Hash` 等 Bean |
| 何时运行 | 启动应用时（`java -jar` 或 IDE 运行）跑一次 |
| 用到的依赖 | spring-boot-starter-web（内嵌 Tomcat、自动配置） |

### 3.5 `FileMd5HashTest`（测试类，test 目录的 `script` 包）

| 方法 | 做什么 | 谁调用它 | 它调用谁 | 用到的依赖 |
|---|---|---|---|---|
| `mutateRealDirectory()` | 直接 `new FileMd5Hash()`，对常量 `REAL_PATH` 指向的**真实目录**跑一遍，打印每个文件前后的 MD5/SHA-256，并断言每个文件都变更成功且新旧 MD5 不同 | JUnit / 手动在 IDE 运行 | `FileMd5Hash.mutateAll(REAL_PATH)` | spring-boot-starter-test（JUnit 5） |

> ⚠️ 该测试会真实修改 `REAL_PATH` 目录下的文件，请指向专门的测试副本。它不经过 HTTP，是直接调业务层。

## 5. 完整执行流程（线上 HTTP 路径，端到端）

```
1. 客户端发起：POST /api/media/mutate?path=D:/photos
        │  （spring-boot-starter-web 的 DispatcherServlet 路由到对应方法）
        ▼
2. ApiController.mutate("D:/photos")
        │  log.info 记录请求；调用业务层
        ▼
3. FileMd5Hash.mutateAll("D:/photos")
        │  3.1 校验路径（不存在/非目录 → 抛 IllegalArgumentException）
        │  3.2 FileUtils.listFiles 递归拿到所有文件        ← commons-io
        │  3.3 for 每个文件：
        │        classify(名字)                            ← commons-io 取扩展名
        │          ├─ 非媒体 → 跳过
        │          └─ 媒体 → mutateOne(file, type)
        │                ├─ FileUtils.readFileToByteArray   ← commons-io 读
        │                ├─ DigestUtils.md5Hex/sha256Hex    ← commons-codec 算旧哈希
        │                ├─ stripTrailer  剥旧尾块          ← 纯 JDK
        │                ├─ appendTrailer 追加随机尾块      ← JDK SecureRandom
        │                │   （do-while 确保新 MD5 ≠ 旧 MD5）
        │                ├─ FileUtils.writeByteArrayToFile  ← commons-io 写回
        │                ├─ DigestUtils.*  算新哈希          ← commons-codec
        │                └─ MediaHashResult.builder()...    ← Lombok builder
        │  3.4 log.info 统计成功数
        ▼
4. 返回 List<MediaHashResult> 给 Controller
        ▼
5. Spring（Jackson）把列表序列化成 JSON
        ▼
6. 客户端收到每个文件的 oldMd5/newMd5、oldSha256/newSha256、体积、是否成功
```

---

## 6. 一句话总结调用关系

> **HTTP 客户端 → `ApiController.mutate` → `FileMd5Hash.mutateAll` →（`classify` 筛选 + `mutateOne` 处理，`mutateOne` 内部再调 `stripTrailer`/`appendTrailer`）→ 产出 `MediaHashResult` 列表 → JSON 返回。**
> 业务核心是 `FileMd5Hash`，哈希靠 commons-codec，文件读写遍历靠 commons-io，样板代码靠 Lombok，Web 入口靠 spring-boot-starter-web。
